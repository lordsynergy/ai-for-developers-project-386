# Проект: Календарь звонков

## Описание проекта

Проект представляет собой упрощённый аналог Cal.com — сервиса для бронирования времени на встречу.

В приложении есть один владелец календаря и гости. Владелец календаря настраивает типы встреч и своё рабочее время.
Гости открывают публичную страницу, выбирают тип встречи, смотрят свободные слоты в ближайшие 14 дней и бронируют подходящее время.

После бронирования выбранное время становится занятым и больше не должно отображаться как доступное для других гостей.

Проект намеренно ограничен по функциональности. В нём не нужны внешние интеграции, синхронизация с Google Calendar,
несколько владельцев календарей, команды, платежи, email-уведомления и сложная система ролей.

Цель проекта — пройти полный цикл разработки небольшого веб-приложения в подходе Design First: 
сначала зафиксировать поведение системы и API-контракт, затем по этому контракту отдельно реализовать frontend и backend,
покрыть основные сценарии тестами и подготовить приложение к запуску в Docker-контейнере.

## Подход Design First

Проект выполняется в подходе Design First.

Это значит, что сначала нужно описать внешнее поведение системы и API-контракт, а только потом переходить к реализации frontend и backend.

API-контракт является единым источником правды для обеих частей приложения:

- frontend знает, какие endpoint'ы вызывать и какие данные ожидать;
- backend знает, какие endpoint'ы реализовать и какие данные возвращать;
- ИИ-агенты могут работать независимо и не анализировать внутреннюю реализацию соседней части приложения.

На первом этапе нельзя реализовывать backend, frontend, базу данных, Rails-модели, контроллеры или UI.
Нужно только описать поведение системы и подготовить TypeSpec-спецификацию.

## Роли

### Владелец календаря / админ

В системе есть один владелец календаря.

Владелец календаря считается админом приложения. Он должен иметь доступ к защищённой административной части.

Для учебного проекта не нужна полноценная регистрация пользователей. Достаточно предусмотреть простую админскую авторизацию:
один заранее заданный админ, данные которого могут быть настроены через переменные окружения или другой конфигурационный механизм.

Админ может:

- войти в административную часть;
- посмотреть данные своего профиля;
- создать тип встречи;
- посмотреть список типов встреч;
- изменить или удалить тип встречи, если это предусмотрено контрактом;
- настроить своё рабочее время;
- посмотреть список предстоящих бронирований.

### Гость

Гость не регистрируется и не входит в систему.

Гость может:

- открыть публичную страницу;
- посмотреть доступные типы встреч;
- выбрать тип встречи;
- посмотреть свободные слоты для выбранного типа встречи в ближайшие 14 дней;
- создать бронирование на свободный слот, указав своё имя и email.

## Основные сущности

### Owner

Владелец календаря.

Ожидаемые поля:

- id;
- name;
- timezone.

Timezone нужен для корректного расчёта рабочих часов и окна бронирования.

### EventType

Тип встречи, которую можно забронировать.

Ожидаемые поля:

- id;
- title;
- description;
- durationMinutes.

Примеры типов встреч:

- Intro call на 30 минут;
- Consultation на 60 минут;
- Demo call на 45 минут.

### AvailabilityRule

Правило рабочего времени владельца календаря.

Рабочее время задаётся недельными правилами.

Ожидаемые поля:

- id;
- dayOfWeek;
- startTime;
- endTime.

dayOfWeek задаётся числами от 1 до 7:

- 1 — Monday;
- 2 — Tuesday;
- 3 — Wednesday;
- 4 — Thursday;
- 5 — Friday;
- 6 — Saturday;
- 7 — Sunday.

startTime и endTime задаются как локальное время в timezone владельца, например:

- 10:00;
- 18:00.

### Slot

Свободный слот для бронирования.

Слоты не обязательно являются отдельной сущностью в базе данных. Они могут вычисляться сервером на основе:

- типа встречи;
- длительности встречи;
- timezone владельца;
- рабочего времени владельца;
- уже существующих бронирований;
- окна записи на ближайшие 14 дней.

Ожидаемые поля:

- eventTypeId;
- startAt;
- endAt;
- durationMinutes;
- available.

### Booking

Бронирование встречи гостем.

Ожидаемые поля:

- id;
- eventTypeId;
- eventTypeTitle;
- guestName;
- guestEmail;
- startsAt;
- endsAt;
- createdAt.

## Правила бронирования

Система должна формировать свободные слоты на ближайшие 14 дней, начиная с текущей даты сервера.

Гость может создать бронирование только на свободный слот из этого окна.

Свободный слот должен соответствовать рабочему времени владельца.

Нельзя создать бронирование вне рабочего времени владельца.

Нельзя создать бронирование, которое пересекается по времени с уже существующим бронированием.

Это правило действует даже если бронирования относятся к разным типам встреч.

Например, если уже есть бронирование:

```text
10:00–11:00
```
то нельзя создать новое бронирование:

```
10:30–11:00
```

или:

```
10:30–11:30
```

## Публичные сценарии гостя

### Просмотр типов встреч

Гость открывает публичную страницу и получает список доступных типов встреч.

Для каждого типа встречи доступны:

- id;
- title;
- description;
- durationMinutes.

### Просмотр свободных слотов

Гость выбирает тип встречи и получает список свободных слотов на ближайшие 14 дней.

Слоты должны быть рассчитаны сервером.

Frontend не должен самостоятельно определять, какие слоты доступны.

### Создание бронирования

Гость выбирает свободный слот и отправляет данные бронирования.

Гость должен указать:

- eventTypeId;
- startAt;
- guestName;
- guestEmail.

Если слот свободен и входит в допустимое окно записи, система создаёт бронирование.

Если слот уже занят, система должна вернуть ошибку конфликта.

Если слот не входит в окно записи, не соответствует рабочему времени или не является допустимым слотом, система должна вернуть ошибку валидации.

## Административные сценарии

### Вход в админку

Админ должен иметь возможность войти в административную часть.

Для учебного проекта достаточно простой схемы:

- один заранее заданный админ;
- логин и пароль задаются конфигурацией;
- после успешного входа backend возвращает token;
- защищённые admin endpoint'ы требуют `Authorization: Bearer <token>`.

### Управление типами встреч

Админ должен иметь возможность создавать типы встреч.

Минимально необходимый сценарий:

- создать тип встречи;
- получить список типов встреч.

Дополнительно можно предусмотреть:

обновить тип встречи;
удалить тип встречи.

### Настройка рабочего времени

Админ должен иметь возможность задать недельные правила рабочего времени.

Например:
```
Monday    10:00–18:00
Tuesday   10:00–18:00
Wednesday 10:00–18:00
Thursday  10:00–18:00
Friday    10:00–18:00
```

Правила рабочего времени должны использоваться при расчёте свободных слотов.

### Просмотр предстоящих бронирований

Админ должен видеть список будущих бронирований всех типов встреч.

Список должен быть отсортирован по времени начала встречи по возрастанию.

## API-контракт

API-контракт должен быть описан через TypeSpec.

TypeSpec-спецификация должна зафиксировать:

- модели данных;
- публичные endpoint'ы гостя;
- защищённые endpoint'ы админа;
- формат запросов;
- формат ответов;
- основные ошибки;
- правила бронирования в комментариях к операциям.

Ожидаемые группы endpoint'ов:

### Public API
- получить публичный список типов встреч;
- получить свободные слоты для выбранного типа встречи;
- создать бронирование.

### Admin API
- создать админскую сессию;
- завершить админскую сессию;
- получить профиль владельца;
- получить список типов встреч;
- создать тип встречи;
- обновить тип встречи;
- удалить тип встречи;
- получить правила рабочего времени;
- обновить правила рабочего времени;
- получить список предстоящих бронирований.

### Ошибки API

В контракте нужно описать основные ошибки:

- 400 Bad Request — некорректный запрос;
- 401 Unauthorized — не передан или неверен admin token;
- 404 Not Found — сущность не найдена;
- 409 Conflict — выбранный слот уже занят;
- 422 Validation Error — данные корректны синтаксически, но нарушают бизнес-правила.

Примеры бизнес-ошибок для 422:

- слот вне окна ближайших 14 дней;
- слот вне рабочего времени владельца;
- startAt не соответствует допустимому свободному слоту;
- правило рабочего времени имеет startTime позже endTime;
- правила рабочего времени пересекаются между собой.
- Что не входит в первую версию

В первую версию проекта не входят:

- регистрация пользователей;
- несколько владельцев календарей;
- роли и права доступа;
- интеграция с Google Calendar;
- email-уведомления;
- платежи;
- команды;
- отмена бронирования гостем;
- перенос бронирования;
- исключения по конкретным датам;
- отпуска и праздничные дни;
- буферы между встречами;
- видеоконференции;
- сложная настройка availability.

Эти возможности можно добавить позже, но они не должны усложнять первый API-контракт.

## Технологии

Проект может быть реализован на любом стеке.

Планируемая реализация backend — Ruby on Rails.

Однако TypeSpec-контракт не должен быть привязан к Rails, ActiveRecord, PostgreSQL, контроллерам или внутренней структуре backend-приложения.

Контракт должен описывать только внешнее поведение API.

Итоговое приложение должно собираться в Docker-образ и запускаться в контейнере.

## Первый этап работы агента

На первом этапе агент должен подготовить TypeSpec-спецификацию.

Агент не должен:

- реализовывать backend;
- реализовывать frontend;
- создавать Rails-приложение;
- писать миграции;
- создавать модели;
- писать UI;
- добавлять Dockerfile.

Агент должен:

- изучить это описание проекта;
- выделить доменные модели;
- подготовить TypeSpec-спецификацию;
- описать публичные и admin endpoint'ы;
- описать основные ошибки;
- добавить комментарии к операциям, где фиксируются бизнес-правила;
- при необходимости добавить tspconfig.yaml для генерации OpenAPI.

Результатом первого этапа должен быть API-контракт, который можно использовать как единый источник правды для будущей реализации frontend и backend.

## Второй этап работы агента: frontend

На втором этапе нужно реализовать пользовательский интерфейс приложения на основе API-контракта из первого этапа.

Frontend должен быть отдельной частью приложения и не должен зависеть от внутренней реализации backend.

Планируемый стек frontend:

- React;
- TypeScript;
- Vite;
- shadcn/ui или простые собственные UI-компоненты;
- API client, работающий с backend через HTTP;
- Prism mock server для разработки frontend без готового backend.

Backend на этом этапе реализовывать нельзя.

Rails-приложение, модели, миграции, контроллеры и база данных на этом этапе не создаются.

Frontend должен использовать только API, описанный в TypeSpec/OpenAPI-контракте:

- `spec/main.tsp`;
- `spec/openapi/openapi.yaml`.

Если frontend требует данных или действий, которых нет в контракте, нужно сначала обсудить изменение контракта, а не добавлять произвольные endpoint'ы.

### Цель второго этапа

Собрать рабочий UI приложения, который можно запускать отдельно и подключать к API по контракту.

Frontend должен корректно работать:

- с mock API, поднятым через Prism по `spec/openapi/openapi.yaml`;
- с настоящим backend, когда он будет реализован позже.

API base URL должен настраиваться через переменную окружения:

```text
VITE_API_BASE_URL
```

Например:
```
VITE_API_BASE_URL=http://localhost:4010
```
для Prism mock API, или:

```
VITE_API_BASE_URL=http://localhost:3000
```

для будущего Rails backend.

### Структура frontend

Frontend нужно разместить в отдельной директории:

`frontend/`

Ожидаемая структура может быть такой:

```
frontend/
  package.json
  index.html
  src/
    main.tsx
    App.tsx
    api/
      client.ts
    pages/
    components/
```

Конкретная структура может отличаться, если она остаётся понятной и не смешивает frontend с backend.

## Public UI

Frontend должен реализовать публичную часть для гостя.

Гость не авторизуется.

Минимальные страницы публичной части:

1. Страница списка типов встреч.

На странице должны отображаться доступные типы встреч:
```
title;
description;
durationMinutes.
```

Данные загружаются через:

`GET /api/public/event-types`

2. Страница выбора слота для выбранного типа встречи.

На странице должны отображаться свободные слоты для выбранного типа встречи.

Данные загружаются через:

`GET /api/public/event-types/{eventTypeId}/slots`

Frontend не должен самостоятельно рассчитывать свободные слоты. Он только показывает слоты, полученные от API.

Форма создания бронирования.

Гость выбирает слот и вводит:
- guestName;
- guestEmail.

Бронирование создаётся через:

`POST /api/public/bookings`

При успешном бронировании frontend должен показать понятное подтверждение.

При ошибках frontend должен показать понятное сообщение:

- слот уже занят;
- слот вне допустимого окна;
- некорректные данные;
- тип встречи не найден.

## Admin UI

Frontend должен реализовать административную часть для владельца календаря.

Админская часть защищена Bearer token.

Минимальные страницы admin-части:

1. Страница входа в админку.

Админ вводит:

- email;
- password.

Запрос выполняется через:

`POST /api/admin/sessions`

После успешного входа frontend сохраняет token в `localStorage`.

2. Страница списка типов встреч.

Данные загружаются через:

`GET /api/admin/event-types`

Запрос должен отправляться с заголовком:

`Authorization: Bearer <token>`

3. Форма создания типа встречи.

Админ вводит:

- id;
- title;
- description;
- durationMinutes.

Запрос выполняется через:

`POST /api/admin/event-types`

4. Форма редактирования типа встречи.

Запрос выполняется через:

`PATCH /api/admin/event-types/{eventTypeId}`

5. Удаление типа встречи.

Запрос выполняется через:

`DELETE /api/admin/event-types/{eventTypeId}`

6. Страница настройки рабочего времени.

Данные загружаются через:

`GET /api/admin/availability-rules`

Сохранение выполняется через:

`PUT /api/admin/availability-rules`

UI должен позволять настроить недельные правила рабочего времени:

- dayOfWeek;
- startTime;
- endTime.

7. Страница предстоящих бронирований.

Данные загружаются через:

`GET /api/admin/bookings/upcoming`

Список должен показывать будущие бронирования всех типов встреч, отсортированные по времени начала.

8. Logout.

Выход выполняется через:

`DELETE /api/admin/sessions`

После выхода token удаляется из localStorage.

## Работа с API

Все HTTP-запросы должны быть вынесены в отдельный API client.

Frontend-компоненты не должны напрямую собирать fetch-запросы в разных местах приложения.

API client должен:

- использовать VITE_API_BASE_URL;
- отправлять JSON;
- обрабатывать успешные ответы;
- обрабатывать ошибки API;
- подставлять `Authorization: Bearer <token>` для admin-запросов.

## Работа с Prism

Для разработки frontend без backend можно использовать Prism mock server.

Пример запуска из корня проекта:

`npx @stoplight/prism-cli mock spec/openapi/openapi.yaml --port 4010`

После этого frontend можно запускать с:

`VITE_API_BASE_URL=http://localhost:4010`

Frontend должен работать с этим mock API без необходимости запускать backend.

### Что нельзя делать на втором этапе

На втором этапе агент не должен:

- реализовывать backend;
- создавать Rails-приложение;
- создавать Rails-модели;
- писать миграции;
- писать backend-контроллеры;
- менять TypeSpec/OpenAPI-контракт без отдельного согласования;
- добавлять endpoint'ы, которых нет в контракте;
- рассчитывать свободные слоты на frontend;
- хранить бизнес-логику бронирования на frontend.

Frontend отвечает только за отображение данных, формы и вызовы API.

### Результат второго этапа

Результатом второго этапа должен быть frontend в директории `frontend/`.

Frontend должен:

- запускаться локально;
- отображать публичные страницы гостя;
- отображать admin-страницы владельца;
- работать с API через `VITE_API_BASE_URL`;
- использовать endpoint'ы из `spec/openapi/openapi.yaml`;
- корректно обрабатывать основные ошибки API;
- не содержать backend-реализации


## Третий этап работы агента: backend

На третьем этапе нужно реализовать backend приложения на основе API-контракта из первого этапа.

Backend должен быть отдельной частью приложения и должен предоставлять HTTP API, описанный в TypeSpec/OpenAPI-контракте:

- `spec/main.tsp`;
- `spec/openapi/openapi.yaml`.

Реализация backend должна идти от контракта, а не от выбранного фреймворка.

Если в процессе реализации окажется, что для backend не хватает endpoint'а, поля или бизнес-ошибки, нельзя молча добавлять произвольное поведение.
Нужно явно зафиксировать, что контракт требует изменения, и только после этого менять TypeSpec/OpenAPI.

Планируемый стек backend:

- Ruby on Rails 8;
- Rails API mode;
- SQLite;
- ActiveRecord;
- RSpec для тестов;
- rack-cors для работы с отдельным frontend-приложением.

Backend должен быть полноценным Rails API-приложением с постоянным хранением данных в базе данных.

Несмотря на то, что в исходном учебном задании допускается хранение данных в памяти, в этом проекте нужно сразу реализовать нормальный backend с базой данных, миграциями, моделями и тестами.

Данные не должны теряться после перезапуска backend.

Для этого этапа нужно использовать SQLite.

PostgreSQL на этом этапе не нужен, потому что в первой версии проекта нет задач, ради которых стоит усложнять локальную инфраструктуру:

- нет нескольких владельцев календарей;
- нет высокой конкурентной нагрузки;
- нет сложной отчётности;
- нет фоновых задач;
- нет production-требований к масштабированию;
- нет интеграций с внешними календарями.

При этом backend должен быть написан так, чтобы при необходимости базу можно было заменить на PostgreSQL позже без переписывания бизнес-логики.

### Цель третьего этапа

Получить рабочий backend, который реализует API по контракту и содержит основную бизнес-логику бронирования.

Backend должен:

- предоставлять публичные endpoint'ы для гостя;
- предоставлять защищённые admin endpoint'ы;
- хранить данные в SQLite;
- использовать ActiveRecord-модели и миграции;
- рассчитывать свободные слоты на ближайшие 14 дней;
- создавать бронирования;
- запрещать бронирование занятого или недопустимого слота;
- возвращать ошибки в формате, описанном в контракте;
- содержать seed-данные для локальной разработки;
- содержать минимальные тесты основной бизнес-логики.

Frontend должен иметь возможность работать с этим backend через:

```text
VITE_API_BASE_URL=http://localhost:3000
```

### Структура backend

Backend нужно разместить в отдельной директории:

```text
backend/
```

Ожидаемая структура может быть такой:

```text
backend/
  Gemfile
  config/
  app/
    controllers/
    models/
    services/
    serializers/
  db/
    migrate/
    seeds.rb
  spec/
```

Конкретная структура может отличаться, если она остаётся понятной и не смешивает backend с frontend.

Backend не должен изменять frontend-код, кроме случаев, когда нужно обновить документацию по запуску или переменные окружения.

## Хранение данных

На этом этапе данные должны храниться в SQLite через ActiveRecord.

Нужно создать Rails-модели и миграции для основных сущностей приложения:

- Owner;
- EventType;
- AvailabilityRule;
- Booking;
- AdminSession, если token'ы хранятся в базе.

Минимальные связи:

- Owner имеет много EventType;
- Owner имеет много AvailabilityRule;
- EventType имеет много Booking;
- Booking относится к EventType.

Так как в первой версии проекта есть только один владелец календаря, backend может работать с единственной Owner-записью.

При этом модель Owner всё равно нужна, чтобы не зашивать данные владельца в код и не усложнять будущие изменения.

Backend должен хранить в базе данных:

- владельца календаря;
- типы встреч;
- правила рабочего времени;
- бронирования;
- admin-сессии или token'ы, если они нужны для выбранной реализации авторизации.

После перезапуска backend данные должны сохраняться.

Для локальной разработки нужно добавить seed-данные:

- одного владельца календаря;
- несколько типов встреч;
- рабочее время на будние дни;
- пустой список бронирований.

Пример начальных данных:

```text
Owner:
  name: Demo Owner
  timezone: Europe/Stockholm

Event types:
  Intro call — 30 minutes
  Consultation — 60 minutes

Availability:
  Monday-Friday 10:00-18:00
```

Бизнес-логику нельзя размазывать по контроллерам.

Контроллеры должны отвечать только за HTTP-слой:

- принять запрос;
- вызвать нужный сервис;
- вернуть JSON-ответ;
- вернуть корректную ошибку.

Расчёт слотов, проверку пересечений бронирований и валидацию бизнес-правил нужно вынести в отдельные сервисы или доменные объекты.

## Public API

Backend должен реализовать публичные endpoint'ы из контракта.

### Получение публичного списка типов встреч

Endpoint:

```text
GET /api/public/event-types
```

Backend должен вернуть список доступных типов встреч.

Ответ должен соответствовать контракту.

### Получение свободных слотов

Endpoint:

```text
GET /api/public/event-types/{eventTypeId}/slots
```

Backend должен:

- найти тип встречи по `eventTypeId`;
- взять длительность встречи;
- рассчитать свободные слоты на ближайшие 14 дней;
- учитывать timezone владельца;
- учитывать недельные правила рабочего времени;
- исключить слоты, которые пересекаются с уже существующими бронированиями;
- вернуть только доступные слоты.

Frontend не должен рассчитывать слоты самостоятельно.

### Создание бронирования

Endpoint:

```text
POST /api/public/bookings
```

Backend должен принимать:

- eventTypeId;
- startAt;
- guestName;
- guestEmail.

При создании бронирования backend должен самостоятельно вычислить `endsAt` на основе длительности выбранного типа встречи.

Backend не должен доверять frontend в вопросах доступности слота.

Перед созданием бронирования backend должен проверить:

- тип встречи существует;
- `startAt` является корректной датой и временем;
- `guestName` заполнен;
- `guestEmail` заполнен и похож на email;
- слот входит в окно ближайших 14 дней;
- слот соответствует рабочему времени владельца;
- слот соответствует сетке допустимых слотов;
- слот не пересекается с уже существующим бронированием.

Если все проверки пройдены, backend создаёт бронирование и возвращает его в формате контракта.

## Admin API

Backend должен реализовать защищённые admin endpoint'ы из контракта.

### Создание admin-сессии

Endpoint:

```text
POST /api/admin/sessions
```

Backend должен проверить email и password администратора.

Для учебного проекта достаточно одного заранее заданного администратора.

Данные администратора можно задать через переменные окружения:

```text
ADMIN_EMAIL
ADMIN_PASSWORD
```

Если переменные окружения не заданы, можно использовать demo-значения только для локальной разработки.

После успешного входа backend возвращает token.

Token может быть случайным значением, сохранённым в базе через модель `AdminSession`.

### Завершение admin-сессии

Endpoint:

```text
DELETE /api/admin/sessions
```

Backend должен удалить или инвалидировать текущий token.

После logout этот token больше не должен давать доступ к admin endpoint'ам.

### Защита admin endpoint'ов

Все admin endpoint'ы, кроме создания сессии, должны требовать заголовок:

```text
Authorization: Bearer <token>
```

Если token отсутствует, неверен или уже удалён, backend должен вернуть:

```text
401 Unauthorized
```

### Получение профиля владельца

Endpoint:

```text
GET /api/admin/owner
```

Backend должен вернуть данные владельца календаря.

### Управление типами встреч

Backend должен реализовать:

```text
GET /api/admin/event-types
POST /api/admin/event-types
PATCH /api/admin/event-types/{eventTypeId}
DELETE /api/admin/event-types/{eventTypeId}
```

При создании и обновлении типа встречи backend должен проверять:

- id заполнен;
- title заполнен;
- durationMinutes больше 0;
- durationMinutes имеет разумное значение;
- id не конфликтует с уже существующим типом встречи при создании.

Если тип встречи не найден, backend должен вернуть:

```text
404 Not Found
```

Если данные нарушают бизнес-правила, backend должен вернуть ошибку валидации.

### Управление рабочим временем

Backend должен реализовать:

```text
GET /api/admin/availability-rules
PUT /api/admin/availability-rules
```

Правила рабочего времени должны задаваться недельными правилами.

Backend должен проверять:

- dayOfWeek находится в диапазоне от 1 до 7;
- startTime и endTime имеют формат локального времени;
- startTime раньше endTime;
- правила одного дня не пересекаются между собой.

Если правила невалидны, backend должен вернуть ошибку валидации.

### Просмотр предстоящих бронирований

Backend должен реализовать:

```text
GET /api/admin/bookings/upcoming
```

Backend должен вернуть будущие бронирования всех типов встреч.

Список должен быть отсортирован по `startsAt` по возрастанию.

## Бизнес-логика бронирования

Основная бизнес-логика должна быть реализована на backend.

Нельзя переносить эти правила во frontend.

Backend отвечает за следующие правила:

- свободные слоты формируются на ближайшие 14 дней;
- слот должен попадать в рабочее время владельца;
- слот должен соответствовать длительности выбранного типа встречи;
- слот не должен пересекаться с существующими бронированиями;
- бронирование одного типа встречи блокирует это время для всех остальных типов встреч;
- нельзя создать бронирование в прошлом;
- нельзя создать бронирование вне окна записи;
- нельзя создать бронирование на несуществующий тип встречи;
- нельзя создать бронирование на произвольное время, если это время не является допустимым слотом.

Например, если есть бронирование:

```text
10:00-11:00
```

то backend должен запретить новые бронирования:

```text
10:00-10:30
10:30-11:00
10:30-11:30
09:30-10:30
```

Это должно работать даже если новое бронирование относится к другому типу встречи.

## Обработка занятого слота

Если гость пытается забронировать слот, который уже занят или пересекается с существующим бронированием, backend должен вернуть:

```text
409 Conflict
```

Пример ошибки:

```json
{
  "error": {
    "code": "slot_already_booked",
    "message": "Selected slot is already booked"
  }
}
```

Важно: эту проверку нужно выполнять именно в момент создания бронирования.

Недостаточно просто убрать занятый слот из списка свободных слотов, потому что между получением списка слотов и отправкой формы другой пользователь может успеть занять это время.

## Ошибки API

Backend должен возвращать ошибки в формате, описанном в контракте.

Минимально нужно обработать:

```text
400 Bad Request
```

Некорректный JSON, отсутствующие обязательные поля или неверный формат запроса.

```text
401 Unauthorized
```

Не передан или неверен admin token.

```text
404 Not Found
```

Запрошенная сущность не найдена.

```text
409 Conflict
```

Выбранный слот уже занят.

```text
422 Validation Error
```

Запрос синтаксически корректен, но нарушает бизнес-правила.

Примеры ситуаций для `422 Validation Error`:

- слот вне ближайших 14 дней;
- слот вне рабочего времени;
- startAt не соответствует допустимому слоту;
- durationMinutes меньше или равно 0;
- правило рабочего времени имеет startTime позже endTime;
- правила рабочего времени пересекаются.

## Тесты backend

Нужно добавить минимальные backend-тесты для основной бизнес-логики.

Тестами нужно покрыть:

- получение списка типов встреч;
- расчёт свободных слотов;
- создание бронирования;
- запрет повторного бронирования того же слота;
- запрет пересекающегося бронирования;
- запрет бронирования вне рабочего времени;
- запрет бронирования вне окна ближайших 14 дней;
- admin login;
- отказ в доступе к admin endpoint'ам без token.

Тесты должны проверять не внутреннюю реализацию Rails, а поведение API и бизнес-правила.

Предпочтительно использовать request specs для проверки HTTP API и отдельные unit/service specs для расчёта слотов и проверки пересечений бронирований.

## CORS

Backend должен быть готов к работе с отдельным frontend-приложением.

Так как frontend запускается отдельно, например на Vite dev server, backend должен разрешать CORS для локальной разработки.

Минимально нужно разрешить frontend origin:

```text
http://localhost:5173
```

Если origin настраивается через переменную окружения, это будет лучше:

```text
FRONTEND_ORIGIN=http://localhost:5173
```

## Проверка результата

После реализации backend должен проходить базовый сценарий вручную или через тесты:

1. Запустить backend.
2. Подготовить базу данных.
3. Загрузить seed-данные.
4. Получить список публичных типов встреч:

```text
GET /api/public/event-types
```

5. Получить свободные слоты для выбранного типа встречи:

```text
GET /api/public/event-types/{eventTypeId}/slots
```

6. Создать бронирование на один из свободных слотов:

```text
POST /api/public/bookings
```

7. Повторно запросить свободные слоты.
8. Убедиться, что забронированный слот больше не возвращается.
9. Попробовать создать бронирование на этот же слот ещё раз.
10. Убедиться, что backend вернул:

```text
409 Conflict
```

11. Войти в admin API.
12. Получить список предстоящих бронирований.
13. Убедиться, что созданное бронирование отображается в admin-части.
14. Перезапустить backend.
15. Убедиться, что данные не пропали, потому что они хранятся в SQLite.

## Что нельзя делать на третьем этапе

На третьем этапе агент не должен:

- реализовывать frontend;
- менять frontend-страницы без необходимости;
- добавлять endpoint'ы, которых нет в TypeSpec/OpenAPI-контракте;
- менять контракт без явного обоснования;
- хранить данные в памяти вместо SQLite;
- обходить ActiveRecord и хранить состояние в глобальных переменных;
- подключать PostgreSQL без отдельной необходимости;
- добавлять регистрацию пользователей;
- добавлять несколько владельцев календарей;
- добавлять Google Calendar;
- добавлять email-уведомления;
- добавлять оплату;
- переносить бизнес-логику бронирования во frontend.

## Результат третьего этапа

Результатом третьего этапа должен быть backend в директории:

```text
backend/
```

Backend должен:

- запускаться локально;
- предоставлять API по `spec/openapi/openapi.yaml`;
- работать с отдельным frontend через HTTP;
- использовать Ruby on Rails 8;
- использовать Rails API mode;
- использовать SQLite;
- использовать ActiveRecord;
- содержать Rails-модели и миграции;
- содержать seed-данные для локальной разработки;
- реализовывать публичные сценарии гостя;
- реализовывать admin-сценарии владельца;
- рассчитывать свободные слоты на backend;
- создавать бронирования;
- запрещать бронирование занятого или пересекающегося слота;
- возвращать корректные ошибки API;
- содержать минимальные тесты основной бизнес-логики.

После этого этапа приложение должно иметь:

- готовый API-контракт;
- готовый frontend;
- рабочий Rails 8 backend;
- постоянное хранение данных в SQLite.

Frontend должен иметь возможность работать с настоящим backend через тот же API-контракт и ту же переменную окружения `VITE_API_BASE_URL`, без изменения frontend-кода.
